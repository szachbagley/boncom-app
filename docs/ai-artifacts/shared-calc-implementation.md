# Shared Calculation Module — Implementation Plan

**Feature:** Make the pure estimate-calculation module usable by the frontend for live
totals preview, using the EXACT same logic the backend uses authoritatively — so the live
preview and the server-of-record can never disagree.
**Branch:** `feat/shared-calc`
**Touches:** `server/src/calculations/` (a one-line banner on the canonical file) and
`client/src/calculations/` (the mirror + guard). No deploy config, no service/route/UI
changes.

Written for straightforward execution — every file, its exact provenance, and every
verification step is specified below.

---

## The architecture decision (confirmed with the user; to be logged in DECISIONS.md)

**Chosen: one canonical source + an automated byte-equality drift guard.** NOT true code
sharing via workspaces.

**Why not true sharing.** This repo is two independent packages (`client/`, `server/`) with
no npm workspace, and **each deploys from its own subdirectory** — Railway builds `server/`
(`server/railway.json`), Vercel builds `client/`. Every genuinely-shared mechanism (npm
workspaces, a root-level `shared/` package, a `file:` dependency) places the shared code
*outside both deploy roots*, which would require reconfiguring both Railway and Vercel (root
directories, build/install commands, Vercel's "include files outside root" toggle) —
deploy-infra surgery on two platforms that can't be verified from here, against the stated
time constraints. The server's `tsc` config (`rootDir: "src"`, fixed `dist/server.js` start
path) further resists a neutral shared location.

**Why the guard makes the mirror safe.** The prompt's real concern with mirroring is *drift*
("duplicated, drift-prone math"). We remove the drift, not just warn about it: an automated
test in **both** suites reads the two copies and **fails the instant they differ byte-for-
byte**. A divergence can't reach `main` green. The module is a 120-line, zero-import, zero-
dependency pure function — maximally safe to mirror. This is deploy-trivial (each subdir
stays self-contained) and a clean future migration to workspaces stays open if desired.

---

## What makes this low-risk (established facts)

- The calc module has **zero imports** — fully self-contained. A byte copy is a complete,
  correct module with nothing to rewire.
- Both toolchains resolve a `.js`-extension import to the `.ts` file: server via NodeNext,
  client via `moduleResolution: bundler` (Vitest uses the same). So the module AND its test
  file can be **byte-identical** copies — the test's `import { … } from './estimate.js'`
  works verbatim in the client too. Byte-identical is what lets a simple equality check be
  the guard.
- The client tsconfig is *stricter* than the server's (`noUnusedLocals`,
  `noUnusedParameters`). The calc module has no unused locals/params (verified), so it
  compiles clean under the client config — but this is checked explicitly in Phase 1.

## Design decisions (baked in; flagged for visibility)

1. **`server/src/calculations/estimate.ts` is THE canonical source of record.** The server
   computes the authoritative totals; its copy is the one humans edit. The client copy is
   generated by copying it, never hand-edited.

2. **A one-line mirror banner is added to BOTH copies** (identically, preserving byte-
   equality). It sits at the top of the file — where a developer about to edit actually
   looks — and names the canonical location, the "don't hand-edit the mirror" rule, and the
   guard. Putting it in the canonical file too is correct (the file genuinely is mirrored)
   and is what keeps the two byte-identical. This is the only change to the server file; it
   touches no logic and no tests. *Decision to log: canonical file gets a banner; re-run
   server tests to confirm untouched behavior.*

3. **The test file is mirrored too, not just the module.** The client re-runs the exact same
   28 tests (including the `$59.65` worked example) through its own toolchain — that is the
   concrete proof "the frontend can call `calculateEstimate(input) -> EstimateTotals`," and
   it means the frontend independently demonstrates identical results, not just identical
   source. Satisfies "keeps its existing tests; do not weaken them" — they're preserved on
   the server AND now additionally exercised on the client.

4. **The drift guard lives in BOTH suites** (`estimate.sync.test.ts` in each package), each
   reading the other package's copy and asserting byte-equality of `estimate.ts` and
   `estimate.test.ts`. Redundant on purpose: neither `npm test` (server) nor `npm test`
   (client) can pass while the copies have drifted, so drift is caught regardless of which
   suite CI/a developer runs. *Decision to log.*

5. **`@types/node` is added to the client's devDependencies.** The client drift-guard test
   reads files via `node:fs`/`node:path`/`node:url`, which need Node types to typecheck
   under the client's `tsc -b` (test files are in the client tsconfig's `include`).
   Justification per CLAUDE.md: a dev-only type package required by a test utility; also
   standard in Vite projects (Vite/Vitest configs routinely use Node builtins). No runtime/
   bundle impact — types only, and the guard is a test file Vite never bundles. *Decision to
   log.*

6. **Byte-equality, not AST/semantic equality.** The guard compares raw file contents with
   `===`. Simple, zero-dependency, and strictest possible (even a whitespace change fails,
   forcing a deliberate re-copy). The guard's failure message is the loud signpost:
   "client copy has drifted from the canonical server/src/calculations/estimate.ts — re-copy
   it, do not hand-edit."

7. **Formatting stays out of the calc module (already true, preserved).** The module returns
   integer cents; dollar/percent formatting is only in `client/src/utils/format.ts` from the
   foundation. The mirror copies this property verbatim — nothing to change. The eventual
   live-preview UI (next feature) will call `calculateEstimate` → cents, then `centsToDisplay`
   for rendering.

8. **No barrel/index file.** Client code imports directly:
   `import { calculateEstimate } from '../calculations/estimate'`. Matches how the client's
   `api/`/`utils/` are imported (no barrels).

9. **Known, accepted type duplication (flag for the next feature).** The client now has two
   independent, structurally-identical declarations of `Discount` and `EstimateTotals`:
   `api/types.ts` (the wire contract, mirroring API-REFERENCE.md) and
   `calculations/estimate.ts` (the calc module's own types). They coincide today. The edit-
   form feature will map form state → the calc module's `EstimateCalcInput` and render the
   calc module's `EstimateTotals`; it should import totals/discount types from the calc
   module for preview math and from `api/types.ts` for request/response. Noted so that
   feature doesn't trip over the overlap. Not resolved here.

## Exact content notes

### Mirror banner (identical in both `estimate.ts` copies, placed above the existing file doc-comment)
```ts
// ⚠️ MIRRORED FILE — keep byte-identical across server/ and client/.
// Canonical source of record: server/src/calculations/estimate.ts
// The client copy is a verbatim copy; do NOT hand-edit it. Edit the canonical file, then
// re-copy it to client/src/calculations/estimate.ts. An automated drift guard
// (estimate.sync.test.ts, in both packages) fails if the two copies differ.
```
Both files therefore begin with this identical block, then the existing
`/** Estimate calculation module … */` doc comment and code, unchanged. Byte-equality holds
because the banner is the same in both.

### `estimate.sync.test.ts` (one per package; only the relative path to the *other* copy differs)
Shape (client version shown; server version swaps `server`↔`client` in the path):
```ts
import { describe, it, expect } from 'vitest';
import { readFileSync } from 'node:fs';
import { fileURLToPath } from 'node:url';
import { dirname, resolve } from 'node:path';

/**
 * Drift guard: the estimate calc module is mirrored between server/ and client/ (see this
 * dir's README). These must stay byte-identical; this test fails if they diverge, so no
 * drift can reach main green. Canonical source: server/src/calculations/estimate.ts.
 */
const here = dirname(fileURLToPath(import.meta.url));
const thisCopy = (name: string) => readFileSync(resolve(here, name), 'utf8');
const otherCopy = (name: string) =>
  readFileSync(resolve(here, '../../../server/src/calculations', name), 'utf8');

describe('estimate calc module mirror', () => {
  it('estimate.ts is byte-identical to the canonical server copy', () => {
    expect(thisCopy('estimate.ts')).toBe(otherCopy('estimate.ts'));
  });
  it('estimate.test.ts is byte-identical to the canonical server copy', () => {
    expect(thisCopy('estimate.test.ts')).toBe(otherCopy('estimate.test.ts'));
  });
});
```
The server version reads `resolve(here, '../../../client/src/calculations', name)` and its
message names the client copy as the one that must match canonical. Both assert the same
equality, from both directions.

### `client/src/calculations/README.md`
Short: these files are a **mirror** of `server/src/calculations/` (the canonical source);
the calc module is pure and shared this way because the two packages deploy from separate
subdirectories (no workspace); to change the math, edit the **server** copy and re-copy
`estimate.ts` + `estimate.test.ts` here; `estimate.sync.test.ts` enforces byte-equality in
both suites. One-line re-sync command included:
`cp ../../../server/src/calculations/estimate.ts ../../../server/src/calculations/estimate.test.ts .`
(run from `client/src/calculations/`).

## Phases & steps (one step at a time; verify, then STOP)

### Phase 1 — Canonical banner + mirror the module
1. Add the mirror banner to `server/src/calculations/estimate.ts`. Create
   `client/src/calculations/estimate.ts` as a byte-identical copy (copy the file, so equality
   is guaranteed by construction). Verify: (a) `cd server && npm test` still green (banner
   didn't disturb the 28 tests / the module's consumers); (b) `cd client && npx tsc -b`
   compiles the copied module under the client's stricter config (the real risk check —
   `noUnusedLocals`/`noUnusedParameters`). *Stop.*

### Phase 2 — Mirror the tests, run them on the client
2. Add `@types/node` to client devDependencies (`npm install -D @types/node`). Copy
   `estimate.test.ts` → `client/src/calculations/estimate.test.ts` (byte-identical). Run
   `cd client && npm test` → the mirrored 28 calc tests pass in the client toolchain
   (proves the frontend can call `calculateEstimate(input) -> EstimateTotals`), alongside the
   existing 9 formatter tests. `npx tsc -b` clean. *Stop.*

### Phase 3 — Drift guard (both suites) + prove it actually guards
3. Add `estimate.sync.test.ts` to both packages and the `client/src/calculations/README.md`.
   Verify positive: `cd server && npm test` and `cd client && npm test` both green (guards
   pass — copies match). Verify **negative** (that the guard has teeth): temporarily append a
   space to one copy, run both suites, confirm BOTH drift guards go RED, then revert and
   confirm green again. Report the negative result. *Stop.*

### Phase 4 — Docs & land
4. Append the architecture decision to `docs/ai-artifacts/DECISIONS.md` (the deploy-subdir
   constraint; single-canonical + drift-guard; drift-impossible not drift-prone; workspace
   migration left open). Add the `TESTING.md` "Shared calculation module" section. Commit,
   push, open PR `feat/shared-calc` → `main`; **you merge manually**.

## Verification philosophy for this feature
The "surface" of a pure shared library is the **package boundary** — importing and executing
it through the consuming package's own toolchain. The mirrored `estimate.test.ts` running
under the client's Vitest IS the frontend calling `calculateEstimate` and getting
`EstimateTotals` (28 assertions incl. `$59.65`), so that's the real evidence, not a
throwaway. The live-preview *UI* wiring (form → `EstimateCalcInput` → `centsToDisplay`) is
the next feature; this one only makes the calc callable and drift-proof.

## Explicitly NOT in this feature
The live-preview UI / edit form, any mapping between form state and `EstimateCalcInput`, npm
workspaces migration, and any Railway/Vercel configuration.
